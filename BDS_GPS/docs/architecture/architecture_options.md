# BDS_GPS 架构选项文档

## 1. 需求和约束总结

### 1.1 关键需求
- 设计BDS/GPS卫星导航系统C/A码捕获与跟踪模块
- 支持两颗GPS卫星和两颗BDS卫星
- 系统时钟频率：16.368MHz
- GPS C/A码片速率：1.023MHz（16倍系统时钟）
- BDS C/A码片速率：2.046MHz（8倍系统时钟）
- 捕获算法：相关峰检测
- 跟踪算法：三路C/A码（e、p、l）
- 输出：卫星号(sv_num)和5位循环码

### 1.2 关键约束
- 资源使用：尽量节省FPGA资源
- 功耗：低功耗设计
- 实时处理：每个系统时钟周期处理一个采样点
- 模块化设计：便于集成到更大系统

## 2. 架构选项

### 2.1 选项1：单卫星顺序处理架构

#### 描述
- 顺序处理每颗卫星，一次只处理一颗卫星
- 使用共享的相关器和累加器
- 状态机控制捕获和跟踪流程

#### 数据通路
- 采样数据输入 → C/A码生成 → 相关器 → 累加器 → 阈值比较 → 状态机

#### 控制
- 主状态机：空闲 → 捕获 → 跟踪 → 输出
- 子状态机：相位搜索、相关计算、阈值判断

#### 存储
- 采样数据存储：FIFO或RAM
- C/A码存储：查找表或实时生成
- 相关值存储：寄存器

#### 接口
- 输入：采样数据、时钟、复位
- 输出：卫星号、5位循环码、状态指示

#### CDC/RDC
- 单一时钟域，无CDC问题
- 异步复位，同步释放

#### 延迟
- 捕获延迟：较长（顺序处理）
- 跟踪延迟：中等

#### 吞吐量
- 低（一次处理一颗卫星）

#### PPA
- 资源使用：低
- 功耗：低
- 面积：小

#### 验证影响
- 验证简单，顺序执行
- 测试用例较少

### 2.2 选项2：多卫星并行处理架构

#### 描述
- 并行处理多颗卫星，每颗卫星独立处理
- 每颗卫星有独立的相关器和累加器
- 并行状态机控制

#### 数据通路
- 采样数据输入 → 分发到多个并行通道 → 每个通道独立处理 → 结果汇总

#### 控制
- 每颗卫星独立的状态机
- 全局控制器协调各通道

#### 存储
- 每颗卫星独立的存储
- 共享采样数据存储

#### 接口
- 输入：采样数据、时钟、复位
- 输出：多颗卫星的卫星号和5位循环码

#### CDC/RDC
- 单一时钟域，无CDC问题
- 异步复位，同步释放

#### 延迟
- 捕获延迟：短（并行处理）
- 跟踪延迟：短

#### 吞吐量
- 高（并行处理多颗卫星）

#### PPA
- 资源使用：高
- 功耗：高
- 面积：大

#### 验证影响
- 验证复杂，并行执行
- 测试用例多

### 2.3 选项3：混合架构（时分复用）

#### 描述
- 时分复用处理多颗卫星
- 在特定时间片内处理一颗卫星
- 共享相关器和累加器

#### 数据通路
- 采样数据输入 → 时间片分配 → 共享处理单元 → 结果存储 → 下一个时间片

#### 控制
- 时间片控制器
- 卫星选择器
- 共享处理单元状态机

#### 存储
- 共享采样数据存储
- 每颗卫星的结果存储
- 时间片配置存储

#### 接口
- 输入：采样数据、时钟、复位
- 输出：多颗卫星的卫星号和5位循环码

#### CDC/RDC
- 单一时钟域，无CDC问题
- 异步复位，同步释放

#### 延迟
- 捕获延迟：中等（时间片复用）
- 跟踪延迟：中等

#### 吞吐量
- 中等（时间片复用）

#### PPA
- 资源使用：中等
- 功耗：中等
- 面积：中等

#### 验证影响
- 验证复杂度中等
- 测试用例数量中等

### 2.4 选项4：流水线架构

#### 描述
- 流水线处理采样数据
- 每个流水线级处理一个特定任务
- 高吞吐量，低延迟

#### 数据通路
- 采样数据输入 → 流水线级1（C/A码生成） → 流水线级2（相关计算） → 流水线级3（累加） → 流水线级4（阈值比较） → 输出

#### 控制
- 流水线控制器
- 流水线状态机

#### 存储
- 流水线寄存器
- 采样数据缓冲

#### 接口
- 输入：采样数据、时钟、复位
- 输出：卫星号、5位循环码、流水线状态

#### CDC/RDC
- 单一时钟域，无CDC问题
- 异步复位，同步释放

#### 延迟
- 捕获延迟：低（流水线）
- 跟踪延迟：低

#### 吞吐量
- 高（流水线）

#### PPA
- 资源使用：中等
- 功耗：中等
- 面积：中等

#### 验证影响
- 验证复杂度中等
- 测试用例数量中等

## 4. 推荐选项

### 推荐：选项3：混合架构（时分复用）

#### 理由
1. **资源效率**：共享相关器和累加器，节省FPGA资源
2. **性能平衡**：在资源使用和性能之间取得良好平衡
3. **可扩展性**：可以支持更多卫星，只需调整时间片
4. **实现复杂度**：比并行架构简单，比顺序架构复杂度适中
5. **验证友好**：验证复杂度适中，测试用例数量可控

#### 具体实现建议
- 时间片长度：1ms（一个C/A码周期）
- 卫星处理顺序：GPS1 → GPS2 → BDS1 → BDS2 → 循环
- 共享相关器：一个相关器，通过多路复用器连接到不同卫星的C/A码生成器
- 结果存储：每颗卫星独立的结果寄存器

## 5. 风险和缓解措施

### 5.1 风险
1. **时间片冲突**：多颗卫星同时需要处理
2. **资源争用**：共享资源可能导致性能瓶颈
3. **时序问题**：时间片切换可能导致时序问题

### 5.2 缓解措施
1. **时间片调度**：合理安排时间片，避免冲突
2. **资源预留**：为关键资源预留时间片
3. **时序优化**：优化时间片切换逻辑，确保时序满足
