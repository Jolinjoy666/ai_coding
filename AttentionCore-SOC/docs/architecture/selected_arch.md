# 选定架构: 流水线重叠架构 (Option B)

## 决策

从三个候选架构中选择 **选项 B（流水线重叠架构）**。

## 决策理由

1. **MAC 利用率 72%** 满足 P0 目标（> 70%），而选项 A 仅 60%
2. **面积 182K gates**（+3.4%）仍在 176K gates 预算的合理范围内
3. **功耗 31 mW**（+7%）仍在 29.3 mW 目标的合理范围内
4. **吞吐量 19.9K tokens/s** 满足 P0 目标（> 10K tokens/s）且有较大余量
5. **单层延迟 30.2 μs** 满足 P0 目标（< 50 μs）
6. 流水线硬件自动推进，固件反而比选项 A 更简单
7. 入门版 n_head=2 时，选项 C 的每组 MAC 仅 2×4，效率不如选项 B 的完整 4×4

## 关键架构特征

### 1. 流水线重叠

- QKV 投影输出直接流入 FlashAttention 输入缓冲区
- FlashAttention 输出直接流入残差加法器
- 使用 ping-pong 缓冲区（~2KB）实现阶段重叠
- 节省 ~17% 的延迟

### 2. 硬件自动推进

- 流水线 FSM 控制阶段推进，无需 RISC-V 逐阶段干预
- valid/ready 握手协议保证数据一致性
- RISC-V 只负责初始配置和异常处理
- 中断仅在整层完成时产生

### 3. MAC 阵列共享

- 完整 4×4 MAC 阵列被所有阶段共享
- QKV 投影、FlashAttention、MLP 分时复用
- 流水线重叠时通过仲裁避免冲突

### 4. 存储分 bank

- Weight SRAM: 双端口，支持同时读取下一层权重
- Feature SRAM: A/B bank 交替使用
- KV-Cache SRAM: 独立 bank
- 无访存冲突

## 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 流水线气泡 | 设计合理的缓冲区大小（ping-pong） |
| 数据冒险 | 完善的 valid/ready 握手协议 |
| 面积增加 | 缓冲区用寄存器而非 SRAM |
| SRAM 访问冲突 | 分 bank 设计，A/B 交替 |
| 验证复杂度 | 分阶段验证，先验证单模块再集成 |

## 否决选项摘要

- **选项 A（顺序执行）**: MAC 利用率 60% 不满足 P0 目标，否决
- **选项 C（多头并行）**: 面积 +19%、功耗 +30% 超标，入门版每组 MAC 太小，否决
