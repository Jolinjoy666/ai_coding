# 架构选项 C: 多头并行架构

## 概述

将 MAC 阵列分为两组，支持同时计算两个注意力头，大幅提高吞吐量。代价是面积增加和控制复杂度上升。

## 数据通路

```
         ┌─────────────────────────────────────┐
         │         QKV 投影（共享 MAC）          │
         └──────────────┬──────────────────────┘
                        ↓
         ┌──────────────┴──────────────────────┐
         │                                      │
    ┌────┴────┐                           ┌────┴────┐
    │ Head 0  │                           │ Head 1  │
    │ MAC 2×4 │                           │ MAC 2×4 │
    │ + FA    │                           │ + FA    │
    └────┬────┘                           └────┬────┘
         │                                      │
         └──────────────┬──────────────────────┘
                        ↓
         ┌──────────────┴──────────────────────┐
         │      拼接 + 输出投影（共享 MAC）       │
         └──────────────┬──────────────────────┘
                        ↓
         ┌──────────────┴──────────────────────┐
         │      残差 + LayerNorm + MLP           │
         └─────────────────────────────────────┘
```

- MAC 阵列分为两组 2×4，各自独立计算一个头
- QKV 投影和输出投影仍用完整 4×4 阵列
- 每组头有独立的 FlashAttention Core

## 控制策略

- 硬件调度器管理两组头的并行执行
- 同步屏障：两头都完成后才进行拼接
- RISC-V 配置整体参数，硬件自动调度多头
- 中断在整层完成后产生

## 存储架构

- Weight SRAM (16KB): 双端口，支持两头同时读取不同头的权重
- Feature SRAM (16KB): 分为 4 个 bank，支持两头并行读写
- KV-Cache SRAM (8KB): 分为两个独立区域
- 增加头间同步缓冲区：~1KB

## 接口影响

- APB 单主设备，无变化
- 需要额外的配置寄存器指示头数和并行度
- GPIO 可指示当前计算的头编号

## 性能估算

| 阶段 | 周期数 | 说明 |
|------|--------|------|
| QKV 投影 | 384 | 共享 4×4 MAC |
| FlashAttention (2头并行) | 100 | 两组 2×4 MAC 并行 |
| 拼接+输出投影 | 256 | 共享 4×4 MAC |
| 残差+LN | 100 | |
| MLP | 1024 | |
| 残差+LN | 100 | |
| **单层合计** | ~1964 | |
| **2 层合计** | ~3928 | |

注：虽然 FA 并行度提高，但拼接和输出投影增加了开销。整体延迟与选项 B 相近，但吞吐量提升更明显。

## PPA 权衡

| 指标 | 估算 | 变化 |
|------|------|------|
| 面积 | ~210K gates | +19%（双 FA Core + 分 bank） |
| 功耗 | ~38 mW | +30%（更多并行单元） |
| MAC 利用率 | ~75% | +15% |
| 吞吐量 | ~20.4K tokens/s | +23% |

## 验证影响

- 并行执行需要验证同步正确性
- 两头结果拼接顺序需要验证
- SRAM bank 访问冲突需要覆盖
- 固件复杂度适中（硬件自动调度）
- 验证复杂度显著增加

## 风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 面积超标 | 超出 176K 预算 | 缩小 MAC 阵列或减少 SRAM |
| 功耗超标 | 超出 29mW 目标 | 时钟门控未使用的头 |
| 同步错误 | 计算结果错误 | 完善的屏障机制 |
| SRAM 冲突 | 流水线停顿 | 增加 bank 数或增加缓冲 |
| 验证复杂度 | 开发周期延长 | 分阶段验证 |

## 对应需求

- CR-001 到 CR-005（RISC-V 控制架构）
- CR-010 到 CR-018（Attention 引擎，多头并行版）
- CR-020 到 CR-022（MLP 引擎）
- CR-030 到 CR-033（LayerNorm 和残差）
- OQ-006（多头并行条件：n_head >= 2 且 SRAM 带宽足够）
- RK-003 缓解（进一步提高 MAC 利用率）
