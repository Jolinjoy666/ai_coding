# 架构权衡矩阵

## 评估维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| P0 性能指标满足度 | 30% | MAC 利用率 > 70%, 吞吐量 > 10K tokens/s |
| PPA 面积 | 20% | 目标 ~176K gates |
| PPA 功耗 | 15% | 目标 ~29 mW |
| 验证复杂度 | 15% | 影响开发周期 |
| 可扩展性 | 10% | 参数化扩展到标准版/增强版 |
| 固件复杂度 | 10% | 影响软件开发和调试 |

## 评分（1-5 分，5 分最优）

| 维度 | 权重 | 选项 A (顺序) | 选项 B (流水线) | 选项 C (多头并行) |
|------|------|--------------|----------------|-----------------|
| 性能指标 | 30% | 3 (MAC 60%) | 4 (MAC 72%) | 5 (MAC 75%) |
| 面积 | 20% | 5 (176K) | 4 (182K, +3%) | 3 (210K, +19%) |
| 功耗 | 15% | 5 (29 mW) | 4 (31 mW, +7%) | 3 (38 mW, +30%) |
| 验证复杂度 | 15% | 5 (简单) | 3 (流水线验证) | 2 (并行验证) |
| 可扩展性 | 10% | 3 (直接扩展) | 4 (流水线可配) | 4 (并行度可配) |
| 固件复杂度 | 10% | 4 (简单) | 5 (硬件自动) | 4 (硬件调度) |

## 加权得分

**选项 A (顺序):**
= 0.30×3 + 0.20×5 + 0.15×5 + 0.15×5 + 0.10×3 + 0.10×4
= 0.9 + 1.0 + 0.75 + 0.75 + 0.3 + 0.4
= **4.10**

**选项 B (流水线):**
= 0.30×4 + 0.20×4 + 0.15×4 + 0.15×3 + 0.10×4 + 0.10×5
= 1.2 + 0.8 + 0.6 + 0.45 + 0.4 + 0.5
= **3.95**

**选项 C (多头并行):**
= 0.30×5 + 0.20×3 + 0.15×3 + 0.15×2 + 0.10×4 + 0.10×4
= 1.5 + 0.6 + 0.45 + 0.3 + 0.4 + 0.4
= **3.65**

## 分析

### 选项 A 优势
- 面积和功耗最优
- 验证复杂度最低
- 固件简单，调试容易
- 适合入门版验证目标

### 选项 A 劣势
- MAC 利用率仅 60%，低于 P0 目标（70%）
- 吞吐量 16.6K tokens/s，满足 P0 但余量小

### 选项 B 优势
- MAC 利用率 72%，满足 P0 目标
- 吞吐量 19.9K tokens/s，满足 P0 且有余量
- 面积和功耗增加可控（+3%/+7%）

### 选项 B 劣势
- 流水线验证需要额外工作
- 缓冲区增加了面积

### 选项 C 优势
- MAC 利用率最高（75%）
- 吞吐量最高

### 选项 C 劣势
- 面积增加 19%，超出预算
- 功耗增加 30%，超出目标
- 验证复杂度最高
- 入门版 n_head=2，两头并行后每组 MAC 仅 2×4，规模太小

## 推荐

**选择选项 B（流水线重叠架构）**

理由：
1. MAC 利用率 72% 满足 P0 目标（> 70%）
2. 面积和功耗增加可控（+3%/+7%），仍在预算范围内
3. 流水线硬件自动推进，固件反而更简单
4. 可扩展性好：标准版/增强版可配置流水线深度
5. 入门版 n_head=2 时，选项 C 的每组 MAC 太小（2×4），效率反而不如选项 B 的完整 4×4

被否决选项：
- 选项 A：MAC 利用率 60% 不满足 P0 目标，否决
- 选项 C：面积和功耗超标，且入门版每组 MAC 太小，否决
