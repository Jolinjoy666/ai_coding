# 架构选项 B: 流水线重叠架构

## 概述

在相邻阶段之间引入流水线重叠，当一个阶段的输出准备好后立即启动下一阶段的部分计算，提高 MAC 阵列利用率。

## 数据通路

```
QKV投影 ──────→ FlashAttention ──────→ 残差+LN ──────→ MLP ──────→ 残差+LN
   ↓                    ↓                   ↓            ↓            ↓
 MAC阵列           MAC阵列+Softmax      加法器+LN硬件  MAC阵列+GELU  加法器+LN硬件
      ↘                ↘                     ↘
       流水线缓冲区（ping-pong）
```

- QKV 投影输出直接流入 FlashAttention 的输入缓冲区
- FlashAttention 输出直接流入残差加法器
- 使用 ping-pong 缓冲区实现阶段重叠

## 控制策略

- 硬件 FSM 控制流水线推进
- RISC-V 只负责初始配置和异常处理
- 流水线阶段通过 valid/ready 握手同步
- 中断仅在整层完成时产生（不是每个子阶段）

## 存储架构

- Weight SRAM (16KB): 双端口，支持同时读取下一层权重
- Feature SRAM (16KB): 分为 A/B 两个 bank，交替使用
- KV-Cache SRAM (8KB): 独立 bank
- 增加流水线缓冲区：~2KB 寄存器

## 接口影响

- APB 单主设备，无变化
- 流水线状态可通过 STATUS 寄存器查询
- 中断频率降低（每层一次，不是每子阶段一次）

## 性能估算

| 阶段 | 周期数 | 重叠节省 |
|------|--------|----------|
| QKV 投影 | 384 | — |
| FlashAttention | 200 | -50（与 QKV 重叠） |
| 残差+LN | 100 | -30（与 FA 重叠） |
| MLP | 1024 | — |
| 残差+LN | 100 | -20（与 MLP 重叠） |
| **单层合计** | ~1508 | 节省 ~17% |
| **2 层合计** | ~3016 | |
| **延迟 @100MHz** | ~30.2 μs | |

## PPA 权衡

| 指标 | 估算 | 变化 |
|------|------|------|
| 面积 | ~182K gates | +3.4%（流水线缓冲） |
| 功耗 | ~31 mW | +7%（更多寄存器翻转） |
| MAC 利用率 | ~72% | +12% |
| 吞吐量 | ~19.9K tokens/s | +20% |

## 验证影响

- 流水线握手需要额外验证
- 数据冒险场景需要覆盖
- 缓冲区满/空条件需要测试
- 固件复杂度降低（硬件自动推进）

## 风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 流水线气泡 | 利用率下降 | 设计合理的缓冲区大小 |
| 数据冒险 | 结果错误 | 完善的握手协议 |
| 面积增加 | 超出预算 | 缓冲区用寄存器而非 SRAM |
| 验证复杂度 | 开发周期延长 | 分阶段验证 |

## 对应需求

- CR-001 到 CR-005（RISC-V 控制架构）
- CR-010 到 CR-018（Attention 引擎）
- CR-020 到 CR-022（MLP 引擎）
- CR-030 到 CR-033（LayerNorm 和残差）
- CR-070 到 CR-072（固件架构，简化版）
- RK-003 缓解（提高 MAC 利用率）
